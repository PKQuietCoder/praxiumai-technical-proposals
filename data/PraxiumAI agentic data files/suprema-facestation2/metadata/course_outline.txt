## Module 1: Performance Specifications (9 minutes)

> **Module Overview:** Before you can advocate for FaceStation 2 in any stakeholder conversation, you need to own its performance numbers — not just recall them, but apply them to the specific question being asked. This module covers the three figures that come up most in product discussions: matching speed, group matching speed, and user capacity.

### Section 1.1: Matching Speed and User Capacity (6 minutes)

#### Subsection 1.1.1: Apply Performance Figures to Stakeholder Questions

**Duration:** 3 minutes
**Format:** Annotated spec card with narrated walkthrough (audio + interactive hotspots)

**Content:**
- FaceStation 2 performs **1:4,000 face matches per second** in standard 1:N mode — this is the figure to cite when stakeholders ask whether the device can handle high-throughput entry points like lobby turnstiles or shift-change scenarios
- **Group Matching runs at 1:5,000 matches per second** — a faster mode relevant when users are pre-grouped (e.g., by department or floor), which reduces the search pool and increases throughput; cite this when engineering asks about optimization strategies
- **User capacity is 30,000 users in 1:1 mode and 4,000 users in 1:N mode** — the distinction matters: 1:1 requires a card or PIN to identify the user first, then verifies the face; 1:N searches the full database by face alone
- When a sales stakeholder asks "can this handle our 2,500-person campus?", the answer isn't just "yes" — it's "yes, and here's why: 4,000-user 1:N capacity with 1:4,000 match/sec throughput means authentication won't become a bottleneck at peak hours"
- The 1:1 vs. 1:N distinction is a common source of confusion — always clarify which mode a stakeholder is assuming before citing capacity figures

**Learning Check:** *Scenario — A sales engineer says: "The customer has 3,800 employees and wants face-only authentication at the main entrance. Can FaceStation 2 handle it?"* Select the most accurate and complete response:
- A) "Yes, it supports up to 30,000 users." *(Incomplete — cites wrong mode)*
- B) "It depends — 1:N mode supports 4,000 users, which covers 3,800, and matches at 1:4,000/sec, so throughput at peak entry should be manageable." *(Correct)*
- C) "Yes, matching speed is 1:4,000/sec." *(Incomplete — doesn't address capacity)*
- D) "We'd need to check the datasheet." *(Incorrect — you should know this)*

---

#### Subsection 1.1.2: Distinguish 1:1 vs. 1:N in Deployment Context

**Duration:** 3 minutes
**Format:** Branching scenario (two-path decision tree with feedback)

**Content:**
- In **1:N (identification) mode**, the device searches all enrolled faces to find a match — no card or PIN needed; this is the "walk up and get in" experience customers often want, but it's capped at 4,000 users
- In **1:1 (verification) mode**, the user presents a credential first (card, PIN, or employee ID), and the device confirms the face matches that specific record — this scales to 30,000 users because the search space is always 1
- The practical implication: a 10,000-person enterprise site cannot use face-only 1:N authentication on a single FaceStation 2 device — they'd need either 1:1 mode or a distributed configuration with server-side matching
- When leadership asks "why can't we just do face-only for everyone?", you can explain the 4,000-user ceiling and offer the 1:1 alternative as the path to full-site coverage
- Group Matching (1:5,000/sec) is a performance optimization, not a capacity expansion — it speeds up matching within a defined group, not across the full user database

**Learning Check:** *Quick decision — A customer has 8,000 employees and insists on face-only (no card) authentication. Which statement is accurate?*

- B) FaceStation 2 cannot support face-only authentication for 8,000 users on a single device in 1:N mode — the 1:N ceiling is 4,000 users. *(Correct)*
- C) Group Matching at 1:5,000/sec would solve the capacity problem. *(Incorrect — Group Matching affects speed, not capacity)*

### Section 1.2: Operating Illuminance Range (3 minutes)

#### Subsection 1.2.1: Assess Deployment Environments by Lighting Condition

**Duration:** 3 minutes
**Format:** Interactive scenario with illuminance scale visual

**Content:**
- FaceStation 2 operates across **0 lx to 25,000 lx** — from complete darkness to the brightest indoor environments; this is described as the highest operating illuminance available in its class
- **0 lx** means the device functions in total darkness — the 140 IR LEDs provide the illumination the camera needs, independent of ambient light
- **25,000 lx** covers all standard indoor lighting conditions; for reference, a brightly lit office is typically 300–500 lx, and direct sunlight outdoors can exceed 100,000 lx — so 25,000 lx covers indoor environments but not direct outdoor sun exposure
- When a stakeholder asks "will this work in our underground parking garage with minimal lighting?", the answer is yes — 0 lx operation is supported
- When a stakeholder asks "can we mount this at an outdoor entrance facing south?", the honest answer is: the spec covers up to 25,000 lx, which handles shaded outdoor areas and covered entryways, but direct sunlight exposure exceeding 25,000 lx is outside the supported range — recommend a shaded or covered installation
- Always anchor your suitability assessment to the 0–25,000 lx range; don't make claims beyond what the specification supports

## Module 2: Security Features (6 minutes)

> **Module Overview:** Face recognition security is one of the most scrutinized topics in any product review. When a security engineer or skeptical sales stakeholder challenges whether face authentication can be spoofed, you need to respond with precision — citing specific features and connecting each one to the specific vulnerability being raised.

### Section 2.1: Anti-Spoofing and Security Architecture (6 minutes)

#### Subsection 2.1.1: Counter Face Spoofing Concerns with Specification Evidence

**Duration:** 3 minutes
**Format:** Video (scenario-based: simulated product review meeting with stakeholder challenge)

**Content:**
- FaceStation 2 includes **live face detection** — the device actively distinguishes a real, present face from a static representation; this directly addresses the most common spoofing concern: "what if someone holds up a photo?"
- The mechanism is **IR-based image analysis** — the device uses infrared imaging to detect the three-dimensional, heat-emitting characteristics of a live face; a printed photo or LCD screen does not replicate these properties, so the attempt fails at the sensor level
- This means the device is protected against **both printed image attacks and LCD/screen-based spoofing** — two distinct attack vectors that stakeholders often raise separately
- The third security layer is the **Android 5.0 Lollipop operating system**, which provides a hardened software environment; cite this when stakeholders raise concerns about firmware vulnerabilities or unauthorized software access
- FaceStation 2 also maintains a **high-quality image log** — every authentication attempt is recorded with image data, providing an audit trail for security reviews and incident investigation
- When countering security objections, connect each feature to the specific concern raised — don't list all three features generically; map them to the vulnerability the stakeholder named

**Learning Check:** *A security engineer says: "I've seen demos where someone unlocked a face recognition system using a photo on a phone screen. How does FaceStation 2 prevent that?"* Which response is most complete and accurate?
- A) "FaceStation 2 uses Android 5.0 for enhanced security." *(Incomplete — doesn't address the specific attack vector)*
- B) "Live face detection uses IR-based image analysis to distinguish a real face from an LCD screen — a phone screen won't replicate the IR signature of a live face, so that attack fails at the sensor level." *(Correct — directly addresses the specific concern)*
- C) "The device has live face detection, so spoofing isn't possible." *(Overstates — doesn't explain the mechanism or acknowledge limits)*

---

#### Subsection 2.1.2: Connect Security Features to Stakeholder Vulnerability Concerns

**Duration:** 3 minutes
**Format:** Interactive matching exercise (drag-and-drop: vulnerability concern → security feature → mechanism)

**Content:**
- Practice mapping the three security features to the three most common vulnerability concerns raised in product reviews:
  - **Printed photo attack** → Live face detection → IR-based image analysis detects absence of live face characteristics
  - **LCD/screen-based spoofing** → Live face detection → IR imaging distinguishes screen-emitted light from IR-reflective live skin
  - **Firmware/software vulnerability** → Android 5.0 Lollipop security layer → Hardened OS reduces attack surface at the software level
- The image log feature supports **post-incident forensics** — when a security stakeholder asks "how would we know if someone tried to spoof the system?", the image log is your answer
- A critical discipline: **don't overclaim**. The specification supports these three features. If a stakeholder asks about network-level security or encryption protocols, acknowledge that's outside the scope of what the datasheet specifies and offer to follow up
- In a product review, the sequence matters: name the concern, name the feature, explain the mechanism — this structure demonstrates mastery, not just memorization

### Section 3.1: Multi-RFID Card Compatibility (3 minutes)

#### Subsection 3.1.1: Address Card Compatibility Questions by Frequency and Type

**Duration:** 3 minutes
**Format:** Interactive reference card with filterable card type list

**Content:**
- FaceStation 2 supports **dual-band RFID**: LF at 125kHz and HF at 13.56MHz — this means it can read both legacy low-frequency cards and modern high-frequency smart cards from a single reader
- The **125kHz (LF) band** covers: EM cards and HID Prox cards — the legacy proximity cards still common in older installations
- The **13.56MHz (HF) band** covers: MIFARE, MIFARE Plus, DESFire EV1/EV2, FeliCa, iCLASS SE/SR/Seos, and NFC — the modern smart card ecosystem
- The device "reads all card types that HID multiCLASS supports" — this is a useful shorthand when talking to customers already familiar with HID's multi-technology readers
- **Important note on DESFire EV2:** Support is via backward compatibility with DESFire EV1 — CSN and smart card functions are compatible; be precise about this when engineering asks about EV2 support
- **RF read ranges vary by card type:** MIFARE/DESFire/iCLASS read at ~50mm; EM/HID Prox/FeliCa at ~30mm; BLE at ~50mm — cite these when installation teams ask about reader placement

## Module 4: Model Selection and System Configuration (9 minutes)

> **Module Overview:** This is where specification knowledge becomes product decision-making. ### Section 4.1: FS2-D vs. FS2-AWB Model Selection (3 minutes)

#### Subsection 4.1.1: Select the Right Model Variant for Deployment Requirements

**Duration:** 3 minutes
**Format:** Side-by-side comparison interactive (expandable spec rows with scenario overlay)

**Content:**
- The **FS2-D** supports: 125kHz EM + 13.56MHz MIFARE, MIFARE Plus, DESFire EV1/EV2, FeliCa, NFC — a strong credential set for most standard deployments
- The **FS2-AWB** adds three capabilities the FS2-D does not have: **HID Prox** (125kHz), **iCLASS SE/SR/Seos** (13.56MHz), and **2.4GHz BLE** — plus **Wi-Fi (IEEE 802.11 b/g)**
- The decision rule is straightforward: if the customer requires HID Prox cards, iCLASS credentials, BLE mobile access, or wireless network connectivity, the answer is **FS2-AWB**; if none of those are required, **FS2-D** is the appropriate choice
- **BLE** is increasingly relevant for mobile credential deployments — if a customer wants employees to use smartphones as access credentials, FS2-AWB is the only option
- **Wi-Fi** matters for installations where running Ethernet is impractical — FS2-AWB's IEEE 802.11 b/g support enables wireless network connectivity; FS2-D requires a wired Ethernet connection
- When justifying model selection to a customer or internal stakeholder, always reference the specific credential or connectivity requirement that drove the decision — "we're recommending FS2-AWB because your existing HID Prox infrastructure requires 125kHz HID Prox support, which is only available on that model"

- B) FS2-AWB — it supports both 125kHz HID Prox (for the existing infrastructure) and 2.4GHz BLE (for the Phase 2 mobile credential rollout). *(Correct)*
- C) Either model — both support HID Prox. *(Incorrect — HID Prox is FS2-AWB only)*

### Section 4.2: System Configuration Selection (6 minutes)

#### Subsection 4.2.1: Match Deployment Scenarios to System Configurations

**Duration:** 3 minutes
**Format:** Interactive diagram (clickable system configuration map based on the source document flowchart)

**Content:**
- FaceStation 2 supports three system configurations, each suited to different deployment scales and security requirements:
  - **Standalone:** FaceStation 2 connects directly to a door lock and exit button via Relay (1 relay output) and TTL input — simplest configuration, appropriate for single-door or small-site deployments where a central server isn't needed
  - **Standalone-Secure:** FaceStation 2 connects to a **Secure I/O 2 module** via RS-485, which then controls the door lock and exit button — the relay and input connections are moved to the Secure I/O 2, keeping door hardware physically separated from the reader; appropriate when the door controller must be in a secured location
  - **Distributed Access Control:** Multiple FaceStation 2 devices connect to a central **BioStar 2 Server** via TCP/IP, with RS-485 connections to additional readers and relay connections to door hardware — appropriate for multi-door, multi-site, or enterprise deployments requiring centralized management
- The interface specifications that enable these configurations: **RS-485 (1ch Host or Slave, selectable)**, **Relay (1 relay)**, **TTL input (2 inputs)**, **Wiegand (1ch In, 1ch Out)**, **Ethernet (10/100/1000 Mbps)**
- When a customer describes their deployment, listen for three signals: **site size** (single door vs. multi-site), **security requirement** (is the door controller in a secure location?), and **integration need** (does it need to connect to a central access control platform?)

**Learning Check:** *A customer has a single server room door. They want the door lock controller in a locked cabinet, not accessible from the hallway. Which configuration applies?*
- A) Standalone — direct relay connection to the door lock. *(Incorrect — the relay would be accessible at the reader)*
- B) Standalone-Secure — FaceStation 2 connects to a Secure I/O 2 module via RS-485; the Secure I/O 2 controls the door lock from inside the secured cabinet. *(Correct)*
- C) Distributed — requires a BioStar 2 Server. *(Incorrect — overkill for a single door; also doesn't address the physical security requirement)*

#### Subsection 4.2.2: Justify Configuration Recommendations Using Interface Specifications

**Duration:** 3 minutes
**Format:** Case study with guided analysis (written scenario + structured response framework)

**Content:**
- Justifying a configuration recommendation means citing the **specific interface specifications** that make it work — not just naming the configuration type
- For **Standalone:** cite the **1 Relay output** (controls the door lock) and **2 TTL inputs** (connects the exit button and door sensor) — these are the physical connections that make direct door control possible without additional hardware
- For **Standalone-Secure:** cite **RS-485 (1ch Host)** as the connection from FaceStation 2 to the Secure I/O 2 module, and the **Relay on the Secure I/O 2** as the door lock control point — the key security benefit is that the relay (and therefore the door hardware) is physically separated from the reader
- For **Distributed:** cite **Ethernet (10/100/1000 Mbps)** as the TCP/IP backbone connecting devices to BioStar 2 Server, and **RS-485 (Host or Slave, selectable)** for connecting additional readers or slave devices in a daisy-chain topology
- **Wiegand (1ch In, 1ch Out)** is relevant when integrating with legacy access control panels that use Wiegand protocol — cite this when a customer asks about compatibility with their existing panel
- The discipline here: when you recommend a configuration, always follow with "and here's the interface that makes it work" — this is what separates a product owner's recommendation from a sales pitch

- B) "Each FaceStation 2 connects to the BioStar 2 Server via Ethernet (10/100/1000 Mbps TCP/IP), and RS-485 (selectable Host or Slave) connects additional readers or slave devices at each door." *(Correct)*
- C) "The RS-485 interface handles all the connections." *(Incomplete — RS-485 alone doesn't describe the full distributed topology)*

## Course Design Notes

### Pedagogical Approach

This course is designed around **performance-first instructional design** — every subsection is anchored to a job task Alex will perform, not a topic Alex should know. The organizing question for every piece of content is: *"What will Alex do differently in a meeting after learning this?"*

**Scenario-based learning** is used throughout to simulate the cross-functional meeting pressure that defines Alex's role. Rather than presenting specifications as facts to memorize, the course presents them as tools for resolving specific stakeholder challenges. This approach directly addresses Alex's core pain point: translating technical specifications into clear product decisions and stakeholder communication.

**Spaced application** is built into the structure: each subsection introduces a specification, immediately applies it to a realistic scenario, and tests application (not recall) in the learning check. The capstone scenario in the Course Summary requires Alex to synthesize specifications across all six objectives in a single integrated challenge — mirroring the complexity of a real product review meeting.

### Assessment Strategy

**Formative assessments** (learning checks) are embedded in every subsection. They are scenario-based, not definition-based — the correct answer always requires applying a specification to a specific context, not reciting it. Distractors are designed to reflect common errors: citing the wrong mode (1:1 vs. 1:N), overclaiming beyond the specification, or providing technically accurate but contextually incomplete answers.

**Summative assessment** (capstone scenario in Subsection 5.1.1) is a five-part integrated scenario that maps directly to all six learning objectives. It is designed to be completed without reference to course content — simulating the "no datasheet in the meeting" condition specified in Goal G1.

**No knowledge-check questions** test pure recall (e.g., "What is the matching speed?"). Every question requires the learner to apply a specification to a decision or response context.

### Accessibility Considerations

- All video content includes closed captions and a full transcript download
- Interactive diagrams include keyboard navigation and screen-reader-compatible alt text
- Color is never used as the sole differentiator in comparison tables or diagrams (pattern + label redundancy)
- All job aids are available in both PDF (print-optimized) and accessible HTML formats
- The course is optimized for desktop and tablet; mobile is supported but not the primary design target given the complexity of interactive diagrams

### Content Scope Discipline

All content is drawn exclusively from the Suprema FaceStation 2 datasheet [SUPREMA-AHL-FS2-EN-REV10]. Where the source document does not provide information (e.g., BioStar 2 software integration depth, competitive comparisons, ADA compliance certification), the course explicitly acknowledges the boundary of the specification rather than filling gaps with unsupported claims. This discipline is modeled for Alex as a professional practice — knowing the edges of what the spec supports is as important as knowing the spec itself.

## Course Developer Notes

### How This Course Addresses Alex's Pain Points

**Pain Point 1: Serving as the authoritative internal reference without reaching for documentation**

The course is explicitly designed around the condition stated in Goal G1: *"without referencing external documentation."* Every learning check simulates the pressure of being asked a pointed question in a live meeting. The capstone scenario requires Alex to answer five specification questions in sequence — the same cognitive demand as a product review. The job aids are positioned as *post-course* references, not crutches during the course itself, reinforcing the internalization goal.

**Pain Point 2: Aligning diverse stakeholders around a shared understanding of product capabilities**

Each subsection is framed around a specific stakeholder type (sales engineer, security engineer, HR director, facilities manager, leadership) asking a realistic question. This gives Alex practice not just knowing the answer, but knowing *how to frame the answer for the audience asking it*. The security module (Module 2) is particularly attentive to this — the same feature (live face detection) is framed differently for a security engineer (mechanism-focused) vs. a sales stakeholder (outcome-focused).

**Pain Point 3: Model variant confusion (FS2-D vs. FS2-AWB)**

Module 4, Section 4.1 is built around a decision rule framework, not a feature list. Rather than asking Alex to memorize which cards each model supports, the course teaches a decision logic: "Does the deployment require HID Prox, iCLASS, BLE, or Wi-Fi? If yes to any, FS2-AWB. If no, FS2-D." The learning check distractors are specifically designed to catch the most common error: assuming both models support HID Prox.

**Pain Point 4: Translating specs into trade-off rationale**

Subsection 4.2.2 explicitly addresses this — the entire subsection is built around the discipline of following a configuration recommendation with the interface specification that justifies it. The framing ("this is what separates a product owner's recommendation from a sales pitch") speaks directly to Alex's professional identity and the standard expected of someone in a product ownership role.

### SME Input Recommendations

Based on the gaps identified in the learning objectives analysis, the following SME inputs are recommended before finalizing course content:

1. **Stakeholder scenario validation:** Have a sales or pre-sales SME review the stakeholder scenarios in Modules 1–3 to confirm they reflect actual objections and questions encountered in the field
2. **FS2-D vs. FS2-AWB confusion confirmation:** Confirm with the product team whether this distinction is a known source of internal errors — if so, consider expanding Module 4, Section 4.1 to a second subsection with additional practice scenarios
3. **Most-misquoted specifications:** Identify which figures (matching speed, capacity modes, illuminance ceiling) are most frequently cited incorrectly internally — use this to weight the emphasis of learning checks and job aid design
4. **BioStar 2 integration depth:** If Alex is expected to address BioStar 2 integration questions in cross-functional discussions, a supplemental module drawing from BioStar 2 documentation would be required; this is currently out of scope
