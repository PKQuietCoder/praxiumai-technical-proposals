### Section 1.1: Understanding the STM32N6 Product Family (14 minutes)

> This section establishes the foundational architecture of the STM32N6 family and the critical NPU-vs-crypto tradeoff that defines your variant selection decision.

---

#### Subsection 1.1.1: Identify the STM32N6 Variants

**Duration:** 7 minutes
**Format:** Annotated interactive diagram + narrated overview

**Content:**
- The STM32N6 family covers four device lines: STM32N645xx, STM32N647xx (collectively STM32N6x7xx — NPU variants) and STM32N655xx, STM32N657xx (collectively STM32N6x5xx — crypto variants)
- All variants share the same Arm Cortex-M55 core running up to 800 MHz (overdrive), the same communication peripheral set (USB, Ethernet, I2C, SPI, FDCAN), and the same package options (VFBGA142 through VFBGA264)
- The defining split: STM32N6x7xx includes the ST Neural-ART accelerator (NPU); STM32N6x5xx includes the full hardware crypto engine (SAES, CRYP, PKA, MCE)
- No variant in the current family combines both NPU and full crypto — this is the central architectural constraint you must resolve for your product
- The ordering information table (Section 7) encodes this: "7" in the line position = Neural ART option; "5" = No Neural ART option; "5" in the die position = Crypto, 4 Mbytes; "4" = No crypto, 4 Mbytes
- Reading the part number STM32N657X0H3Q: N=neural, 6=Cortex-M55, 5=crypto die, 7=Neural ART line — this combination (N657) is the STM32N6x7xx with NPU but without the crypto die features

### Section 1.2: Making the Variant Selection Decision (18 minutes)

> This section takes you from reading the tables to making and documenting a justified variant selection recommendation, including package selection for your PCB.

---

#### Subsection 1.2.1: Analyze the NPU-vs-Crypto Tradeoff

**Duration:** 7 minutes
**Format:** Branching scenario / decision-tree case study

**Content:**
- A face recognition access control terminal has two non-negotiable requirements: (1) real-time DNN inference for face matching, and (2) secure storage and transmission of biometric templates
- The STM32N6x7xx (NPU variant) provides the Neural-ART accelerator at 600 Gops — essential for running face recognition models in hardware without CPU involvement; it does NOT provide SAES, CRYP, PKA, or MCE
- The STM32N6x5xx (crypto variant) provides SAES (SESIP/PSA Level 3 certified, side-channel attack resistant), CRYP (AES-128/192/256), PKA (public key), and MCE (inline memory encryption) — but NO NPU
- Architectural implication: if you select the NPU variant (x7xx), biometric template encryption must be handled in software on the Cortex-M55 (using Helium vector instructions for AES), or offloaded to a companion security element; if you select the crypto variant (x5xx), face recognition inference must run on the Cortex-M55 CPU without hardware NPU acceleration
- The datasheet does not describe any roadmap variant combining both — treat this as a hard constraint in your architecture document
- Your recommendation must explicitly state this tradeoff and justify which requirement takes priority for your product's performance and certification targets

### Section 2.1: Cortex-M55 Core and On-Chip Memory (14 minutes)

> Understanding the core's capabilities and memory hierarchy is prerequisite to planning your inference pipeline and memory allocation.

---

#### Subsection 2.1.1: Characterize the Cortex-M55 Core

**Duration:** 7 minutes
**Format:** Annotated block diagram + narrated video

**Content:**
- The Cortex-M55 runs at up to 800 MHz in overdrive mode (VOS high) or 600 MHz in nominal mode (VOS low), achieving 3616 CoreMark at 800 MHz (4.52 CoreMark/MHz)
- Arm Helium (M-Profile Vector Extension, MVE) enables SIMD vector processing — critical for software AES if you select the NPU variant without hardware crypto, and for DSP preprocessing of camera frames
- The FPU supports half, single, and double precision (IEEE 754 compliant) — relevant for neural network inference preprocessing and post-processing on the CPU
- TrustZone is integrated into the core with MPU support — this is your hardware isolation boundary between the secure biometric processing domain and the non-secure application domain
- The NVIC supports up to 196 maskable interrupt channels with 16 priority levels — you'll use this to manage DMA completion interrupts from the camera pipeline and NPU
- Cache configuration: 32 KB ICACHE + 32 KB DCACHE — these are separate from TCM and operate on AXI bus accesses; TCM accesses bypass the cache entirely for deterministic latency

**Learning Check:** *True/False* — The Cortex-M55's 32 KB ICACHE and 32 KB DCACHE are the same as the instruction TCM and data TCM. *Answer: False. The ICACHE and DCACHE are AXI bus caches. The ITCM (64 KB) and DTCM (128 KB) are separate tightly-coupled memories with zero-wait-state access that bypass the cache entirely.*

---

#### Subsection 2.1.2: Map the On-Chip Memory Hierarchy

**Duration:** 7 minutes
**Format:** Interactive memory map diagram with annotation tool

**Content:**
- The Cortex-M55 includes 64 KB instruction TCM (ITCM) and 128 KB data TCM (DTCM), both with ECC (single-error correction, double-error detection) — these provide zero-wait-state access for time-critical code and data
- On-chip SRAM includes: AHBSRAM1/2, AXISRAM1 through AXISRAM6, BKPSRAM (backup), FLEXRAM, and VENCRAM — all with ECC featuring single-error detection/correction with interrupt generation and hardware/software erase capability
- AXISRAM1 retains content in Standby mode (per Table 7); AXISRAM2–6 do not retain in Standby — this matters for your access control terminal's wake-from-standby behavior
- The AXI cache (CACHEAXI1) provides an additional caching layer for AXI bus accesses to external memory — relevant when streaming model weights from XSPI-connected flash
- For your face recognition pipeline: ITCM holds time-critical ISR and inference control code; DTCM holds active frame buffer data or intermediate inference results; AXISRAM holds larger working buffers; XSPI-mapped external memory holds model weights and biometric templates
- ECC on all SRAM regions provides functional safety coverage — important for an access control system where memory corruption could cause false accepts or denials

Characteristics: (A) 128 KB, zero-wait-state, ECC, data; (B) Retains content in Standby mode; (C) 64 KB, zero-wait-state, ECC, instruction; (D) Does not retain in Standby mode. *Answers: 1-C, 2-A, 3-B, 4-D.*

### Section 2.2: Power Management for Access Control Applications (7 minutes)

> An access control terminal runs continuously but must manage power intelligently — this section covers the power architecture relevant to your deployment scenario.

---

#### Subsection 2.2.1: Apply Power Management to Your Terminal Design

**Duration:** 7 minutes
**Format:** Scenario-based video + configuration checklist

**Content:**
- The STM32N6 power controller manages four supply domains: Core domain (VDDCORE), VDD domain (VRET), Backup domain (VSW/VBAT), and Analog domain (VDDA18ADC)
- Voltage scaling provides two run-mode settings: VOS high (enables 800 MHz overdrive) and VOS low (600 MHz nominal, lower power) — for a face recognition terminal, you'll likely run VOS high during active inference and consider VOS low during idle monitoring
- The integrated SMPS step-down converter (switching at 2.4 MHz reference frequency) efficiently converts the 1.8V supply to VDDCORE (0.63–0.9V range) — peak efficiency ~88–89% at typical load currents
- Low-power modes available: Sleep (CPU clock stopped, peripherals active), Stop SVOS high (most peripherals retained), Stop SVOS low (reduced retention), Standby — for an access control terminal, Stop SVOS high is the likely idle state, allowing HPDMA and camera to wake the system on motion detection
- The NPU is listed as "O" (optional/configurable) in Run mode and Stop SVOS high, and "R" (retained/off) in Stop SVOS low — plan your power state transitions around NPU availability
- The RTC remains functional in VBAT mode — use this for access log timestamping even during power interruptions

In this mode, HPDMA is retained (R) and can service camera data, but the NPU is listed as "O" (optional — can be kept on or off). If NPU is powered off in Stop SVOS high, you must wake to Run mode before executing inference.*

### Section 3.1: Neural-ART NPU Architecture and Capabilities (14 minutes)

> This section gives you the technical depth to evaluate the NPU against your face recognition workload requirements.

---

#### Subsection 3.1.1: Characterize the Neural-ART NPU Architecture

**Duration:** 7 minutes
**Format:** Annotated block diagram + narrated deep-dive video

**Content:**
- The Neural-ART NPU is available only on STM32N6x7xx devices (Section 3.5) — it is the design-time parametric and runtime reconfigurable neural network inference engine
- Maximum operating frequency: 1 GHz (a dedicated PLL — PLL2 — is provided for the Neural-ART accelerator, separate from the system clock PLL)
- Peak throughput: 600 Giga-operations per second (Gops) — achieved using optimized hardware units for DNN inference functions
- The NPU instantiation includes convolution accelerators (CAs); the specific instantiation can vary, with a low-end version using two CAs
- The host (Cortex-M55) configures the NPU subsystem through an AHB/AXI lite target port connected to the system bus matrix; the NPU has two 64-bit wide AXI-4 controller ports connecting to the system bus for memory access
- System memory is shared between the host subsystem and the NPU subsystem — this means your memory layout must account for concurrent access patterns between the CPU and NPU

**Learning Check:** *Short answer* — What is the NPU's maximum clock frequency, and how does it connect to the system bus? *Answer: 1 GHz maximum, driven by a dedicated PLL. It connects to the system bus matrix via two 64-bit AXI-4 controller ports for memory access, and is configured by the host via an AHB/AXI lite target port.*

---

#### Subsection 3.1.2: Assess NPU Throughput for Face Recognition

**Duration:** 7 minutes
**Format:** Guided assessment worksheet + gap analysis exercise

**Content:**
- Key NPU features for minimizing memory bandwidth bottlenecks: (1) dedicated streaming engines that optimize data flow and minimize internal buffer usage, (2) on-the-fly weight decompression — model weights stored compressed in external flash are decompressed during inference without CPU involvement, (3) real-time data encryption/decryption — the NPU can process encrypted model weights directly
- 600 Gops provides substantial compute headroom for common face recognition model architectures (MobileNet-class models typically require 300–600 MOps per inference at standard input resolutions) — however, the datasheet does not specify per-inference latency in milliseconds for any specific model
- **Critical datasheet gap to flag in your assessment:** The datasheet provides no end-to-end inference latency specification (e.g., milliseconds per face recognition inference at a given model size and input resolution). You cannot verify from the datasheet alone whether 600 Gops achieves your target latency (e.g., <500ms per recognition attempt)
- Your written assessment must cite: 600 Gops at 1 GHz maximum; streaming engines and weight decompression as bandwidth-optimization features; absence of per-inference latency data as a gap requiring characterization on actual hardware with your target model
- Next steps to close the gap: obtain ST's STM32Cube.AI toolchain, profile your target face recognition model (e.g., MobileNetV2-based face embedding network) on the STM32N6 evaluation board, and measure actual inference latency
- The NPU's real-time encryption/decryption capability is relevant even on the x7xx variant — model weights stored in encrypted external flash can be decrypted inline during inference without a separate decryption step

### Section 4.1: Camera Input Subsystem (14 minutes)

> Understanding the camera subsystem's two paths and the CSI-2 interface specifications is the foundation for configuring your face capture pipeline.

---

#### Subsection 4.1.1: Configure the Camera Subsystem Architecture

**Duration:** 7 minutes
**Format:** Interactive pipeline diagram with clickable stage annotations

**Content:**
- The STM32N6 camera subsystem is built around two paths: (1) low-resolution path via DCMI with parallel interface (up to 24 Mpixel/s, 80 MHz on 14-bit input), and (2) high-resolution path via DCMIPP with parallel or CSI-2 serial interface (up to 5 Mpixels at 30 fps)
- For a face recognition terminal, use the DCMIPP path — it connects to the CSI-2 interface and feeds the ISP, which can then feed the NPU via DMA
- The DCMI path is recommended only for backward compatibility or for connecting a second camera sensor (the DCMIPP inputs from one sensor via CSI-2; the DCMI gets pixels from a second sensor via parallel interface)
- The DCMIPP pixel pipeline block includes the parallel interface, the CSI-2 Host, and the PHY — all integrated into a single subsystem
- Parallel interface specifications (for reference): 120 MHz up to 16-bit capability with I/O ReTime; supports RGB565/888, YUV422, RawBayer/Mono 8/10/12/14
- The DCMIPP pixel clock input supports up to 120 MHz with 30–70% duty cycle (per Table 105)

**Learning Check:** *Decision question* — Your face recognition terminal uses a single 5MP RGB camera sensor with a MIPI CSI-2 interface. Which camera path do you use, and why is the DCMI path not appropriate for this configuration? *Answer: Use the DCMIPP path with CSI-2 interface. The DCMI path uses a parallel interface and is recommended only for backward compatibility or a second sensor. The DCMIPP path is designed for high-resolution sensors and connects to the CSI-2 interface, which is required for your MIPI camera.*

---

#### Subsection 4.1.2: Specify the CSI-2 Interface Parameters

**Duration:** 7 minutes
**Format:** Specification table analysis + configuration checklist

**Content:**
- The CSI-2 serial interface complies with MIPI CSI-2 v1.3 (MIPI Alliance Specification, 29 May 2014) and D-PHY v1.2
- Input rate: two data lanes at up to 2.5 Gbps/lane — this gives you a maximum aggregate bandwidth of 5 Gbps, sufficient for high-resolution, high-frame-rate camera sensors
- Pixel rate: up to 200 Mpixel/s for RGB888, up to 333 Mpixel/s for rawBayer 10 — for a 5MP sensor at 30 fps, you need 150 Mpixel/s, well within the 200 Mpixel/s limit
- Supported pixel formats: any MIPI CSI-2 v1.3 format including RGB565, RGB888, YUV422, rawBayer — select rawBayer for maximum ISP flexibility (the ISP can perform demosaicing)
- Additional CSI-2 features: interlaced video support, interleaved packets, virtual channels — virtual channels allow multiplexing multiple logical streams over a single physical CSI-2 link
- The CSI PHY supply current at 2.5 Gbps/lane (2 lanes): approximately 2.54–2.64 mA on VDDA18CSI and 3.41–3.86 mA on VDDCSI — include these in your power budget

**Learning Check:** *Calculation check* — Your camera sensor outputs 5MP (2592×1944) at 30 fps in rawBayer 10-bit format. Calculate the required pixel rate and confirm it is within the CSI-2 interface limit. *Answer: 2592 × 1944 × 30 = 151,165,440 pixels/second ≈ 151 Mpixel/s. The CSI-2 interface supports up to 333 Mpixel/s for rawBayer 10, so 151 Mpixel/s is well within the limit.*

### Section 4.2: ISP Processing and DMA-to-NPU Handoff (7 minutes)

> This section covers the ISP's three-pipe architecture and the critical DMA handoff that delivers processed frames to the NPU without CPU intervention.

---

#### Subsection 4.2.1: Map the ISP-to-NPU Data Path

**Duration:** 7 minutes
**Format:** Animated pipeline flow diagram + written description exercise

**Content:**
- The ISP provides processing of three parallel pipes on the same input stream — all three pipes process the same incoming camera data simultaneously, enabling simultaneous output at different resolutions or formats
- ISP processing algorithms available per pipe: bad pixel correction, decimation, black-level tuning, exposure control, demosaicing, column conversion, contrast adjustment, cropping, downsizing, ROI isolation, gamma correction, YUV conversion, and pixel packer
- **The critical handoff:** The ISP output can be directly fed via a DMA to the NPU — this is the zero-CPU-intervention path for your face recognition pipeline. The CPU does not need to copy frame data; the DMA handles the transfer from ISP output to NPU input buffer
- The HPDMA (High Performance DMA) is the appropriate controller for this transfer — channels 12–15 support 2D addressing and 64-byte FIFOs, suitable for image data transfers including to AXI external memories
- Complete pipeline stage sequence for your firmware architecture document: **CSI-2 PHY** (physical layer, 2.5 Gbps/lane) → **DCMIPP** (pixel pipeline, demultiplexing, format conversion) → **ISP** (three parallel pipes: bad pixel, demosaic, gamma, YUV, crop/resize) → **HPDMA** (DMA transfer, no CPU intervention) → **Neural-ART NPU** (DNN inference, 600 Gops)
- The three ISP pipes enable simultaneous outputs: e.g., Pipe 1 → full-resolution frame for H264 encoding/recording; Pipe 2 → cropped/resized face ROI for NPU inference; Pipe 3 → thumbnail for display

**Learning Check:** *Pipeline description exercise* — Write a four-stage pipeline description for your firmware architecture document, naming each subsystem, its key specification, and the handoff mechanism to the next stage. *Model answer: "(1) CSI-2 interface receives rawBayer 10 data at up to 2.5 Gbps/lane from the camera sensor. (2) DCMIPP pixel pipeline demultiplexes and routes the pixel stream to the ISP. (3) ISP processes three parallel pipes on the same input stream — Pipe 2 crops and resizes the face ROI for inference. (4) HPDMA transfers the ISP output directly to the NPU input buffer without CPU intervention. (5) Neural-ART NPU executes DNN inference at 600 Gops."*

### Section 5.1: Hardware Security Features and Their Roles (14 minutes)

> This section maps each security peripheral to a specific protection function in your biometric terminal, and explicitly addresses the constraint that full crypto is only available on the STM32N6x5xx variants.

---

#### Subsection 5.1.1: Map Security Features to Protection Roles

**Duration:** 7 minutes
**Format:** Interactive security feature mapping table + case study

**Content:**
- **TrustZone (Section 3.1):** Available on all variants. Provides hardware isolation between Secure and Non-Secure worlds. In your terminal: the Secure world runs biometric template storage, matching algorithms, and access decision logic; the Non-Secure world runs the UI, network stack, and logging. TrustZone-aware peripherals (HPDMA, GPIO, RTC) can be individually assigned to Secure or Non-Secure domains
- **SAES — Secure AES Coprocessor (Section 3.33):** Available ONLY on STM32N655xx and STM32N657xx (STM32N6x5xx variants). Encrypts/decrypts data per NIST AES standard. Key differentiator: certified SESIP and PSA security assurance Level 3, with protection against side-channel attacks (SCA) including differential power analysis (DPA). Supports 128/256-bit keys, ECB/CBC/CTR/GCM/GMAC/CCM modes. Role: biometric template encryption/decryption with certified SCA resistance
- **CRYP — Cryptographic Processor (Section 3.34):** Available ONLY on STM32N655xx and STM32N657xx. Higher-throughput AES (128/192/256-bit), supports DMA for bulk data. Role: high-throughput encryption of biometric data streams. Can load keys from SAES via Shared-key mode
- **HASH — Hash Processor (Section 3.35):** Available on all variants (confirmed by Table 7 showing HASH in Run mode). SHA-1, SHA-2 family (SHA-224, SHA-256, SHA-384, SHA-512), HMAC. Role: biometric template integrity verification — hash stored templates and verify on retrieval to detect tampering
- **MCE — Memory Cipher Engine (Section 3.36):** Available ONLY on STM32N6x5xx variants. Provides system bus inline encryption (writes) and decryption (reads) for XSPI-connected external memory. Role: transparent encryption of external flash containing biometric templates — reads and writes are automatically encrypted/decrypted without CPU involvement
- **PKA — Public Key Accelerator (Section 3.37):** Available ONLY on STM32N6x5xx variants. Role: asymmetric cryptography for secure enrollment (signing biometric templates with a device private key) and mutual authentication with the access control server

**Learning Check:** *Table completion* — Complete the security feature mapping table: Feature | Variant Availability | Protection Role. Fill in rows for TrustZone, SAES, MCE, HASH, and PKA. *Model answer: TrustZone | All variants | Secure/Non-Secure domain isolation; SAES | STM32N6x5xx only | Biometric template encryption (SESIP/PSA L3 certified); MCE | STM32N6x5xx only | Inline encryption of XSPI external flash; HASH | All variants | Template integrity verification; PKA | STM32N6x5xx only | Asymmetric crypto for enrollment signing and server authentication.*

---

#### Subsection 5.1.2: Apply MCE Inline Encryption to External Memory

**Duration:** 7 minutes
**Format:** Configuration walkthrough + scenario analysis

**Content:**
- The MCE (Memory Cipher Engine) sits on the system bus between the AXI interconnect and the XSPI interface — all writes to XSPI-mapped external memory pass through MCE for encryption; all reads pass through for decryption
- MCE supports up to 4 configurable regions with 4 KB granularity, each with an independent encryption mode: bypass (no encryption), stream cipher (AES-CTR compliant per NIST SP800-38A), block cipher (AES-ECB compliant per NIST FIPS 197), and fast block cipher
- Supported block ciphers: AES-128, AES-256, or Noekeon (12-round version) — selected at boot time
- MCE includes a leakage-resilient mode of operation as defense against side-channel attacks — relevant for a biometric terminal where physical access by an attacker is a realistic threat model
- For your terminal: configure MCE Region 1 for the biometric template storage area in XSPI-connected NOR flash (stream cipher mode for sequential access efficiency); configure MCE Region 2 for the model weights area (bypass mode if weights are not sensitive, or block cipher if IP protection is required)
- MCE is transparent to the CPU and NPU — the NPU's on-the-fly decryption capability (mentioned in Section 3.5) works in conjunction with MCE for encrypted model weight access
- **Critical constraint:** MCE is absent on STM32N6x7xx (NPU) variants — if you select the NPU variant, external flash encryption must be implemented in software or via a companion secure element

**Learning Check:** *Architecture decision* — You've selected the STM32N6x7xx (NPU variant) for your terminal. A security audit requires that biometric templates stored in external NOR flash must be encrypted at rest. What is your mitigation strategy, given that MCE is not available on this variant? *Answer: Options include: (1) Software AES encryption/decryption on the Cortex-M55 using Helium vector instructions before writing to / after reading from XSPI flash; (2) Use an external secure element (e.g., STSAFE-A) for template encryption and key management; (3) Reconsider variant selection — if the security audit requires hardware-accelerated inline encryption, the STM32N6x5xx (crypto variant) with MCE may be required, accepting the loss of NPU hardware acceleration.*

### Section 6.1: Designing the Memory Layout for Your Inference Pipeline (14 minutes)

> This section integrates your knowledge of the memory hierarchy, NPU requirements, and security constraints into a concrete memory allocation plan.

---

#### Subsection 6.1.1: Allocate On-Chip Memory Regions

**Duration:** 7 minutes
**Format:** Drag-and-drop memory allocation exercise + annotated memory map

**Content:**
- **ITCM (64 KB, ECC):** Place time-critical inference control code here — ISR handlers for DMA completion, NPU job submission routines, and camera frame synchronization code. Zero-wait-state access ensures deterministic latency. Note: 64 KB is sufficient for control code but not for full model inference code; model execution runs from the NPU's own instruction set, not from ITCM
- **DTCM (128 KB, ECC):** Place active frame buffer data for the current inference frame, intermediate preprocessing results, and the face detection bounding box output. Zero-wait-state access from the Cortex-M55 ensures fast preprocessing. The NPU accesses DTCM via the AXI bus (not zero-wait-state for NPU), so consider AXISRAM for NPU input/output buffers
- **AXISRAM1–6 (ECC, various retention characteristics):** Use AXISRAM for NPU input/output feature maps, ISP output staging buffers, and the face embedding vector output. AXISRAM1 retains content in Standby mode — use it for persistent state (e.g., last recognized face ID for quick re-authentication)
- **BKPSRAM:** Use for access log entries and configuration data that must survive power cycles (backed by VBAT)
- **VENCRAM (128 KB):** Available for video encoder use; when VENC is disabled, VENCRAM can be statically assigned to the system via SYSCFG — use as additional working buffer if needed

**Learning Check:** *Allocation decision* — Your NPU inference produces a 512-element float32 face embedding vector (2 KB). Where do you place this output buffer, and why? *Answer: Place in AXISRAM (e.g., AXISRAM2 or AXISRAM3). The NPU accesses AXISRAM via its 64-bit AXI-4 controller ports efficiently. DTCM is optimal for CPU access but the NPU accesses it via AXI (not zero-wait-state for NPU). AXISRAM provides a good balance of NPU and CPU accessibility for the embedding vector that both the NPU (writes) and CPU (reads for matching) need to access.*

---

#### Subsection 6.1.2: Plan External Memory with XSPI and MCE

**Duration:** 7 minutes
**Format:** Memory map diagram + configuration specification exercise

**Content:**
- The XSPI interface supports memory-mapped mode — external NOR flash or HyperRAM appears to the system as internal memory, accessible by address without explicit read commands. This is essential for NPU weight streaming: the NPU's streaming engines can fetch compressed model weights directly from XSPI-mapped flash as if reading internal memory
- XSPI supports single, dual, quad, and octal communication; SDR and DTR (double-transfer rate); HyperBus protocol — for maximum bandwidth to the NPU, use octal DTR mode (up to 200 MHz clock = 400 MT/s effective, 8 bits wide = 400 MB/s peak)
- The XSPI I/O manager (XSPIM) manages three XSPI interfaces (XSPI1, XSPI2, XSPI3) with two physical ports (Port 1: 16-bit I/O; Port 2: 8-bit I/O) — you can connect NOR flash to Port 1 and HyperRAM to Port 2 for simultaneous access
- **Memory allocation for external memory (with MCE on STM32N6x5xx):**
  - MCE Region 1 (stream cipher): Biometric template database in NOR flash — encrypted at rest, decrypted transparently on read
  - MCE Region 2 (bypass or block cipher): Model weights in NOR flash — bypass if not IP-sensitive; block cipher for IP protection
  - MCE Region 3 (bypass): Firmware image and application code — typically not encrypted (verified by secure boot instead)
  - HyperRAM (via XSPI Port 2): Large frame buffers for multi-frame face capture sequences
- **Without MCE (STM32N6x7xx NPU variant):** External flash regions are unencrypted at the hardware level; software encryption must be applied before write and after read for sensitive regions
- Each allocation in your memory specification document must cite the relevant datasheet section: XSPI memory-mapped mode (Section 3.17.1), XSPIM (Section 3.17.2), MCE (Section 3.36), SRAM configuration (Section 3.2)

**Learning Check:** *Specification exercise* — Complete this memory allocation table row: "Model Weights | Location: _______ | Access mode: _______ | Encryption: _______ (STM32N6x5xx) / _______ (STM32N6x7xx) | Datasheet reference: _______." *Model answer: Model Weights | XSPI-mapped NOR flash (memory-mapped mode) | Memory-mapped read (NPU streaming engine fetches directly) | MCE block cipher or bypass (Section 3.36) | Software AES or unencrypted | Section 3.17.1 (XSPI memory-mapped mode), Section 3.36 (MCE).*

## Course Design Notes

### Pedagogical Approach

This course applies a **task-first, reference-second** instructional model. Each subsection begins with the engineering task Jin-ho needs to perform (e.g., "write a variant selection recommendation," "map the pipeline stages"), then provides the datasheet content needed to complete that task. This approach mirrors how experienced engineers actually use datasheets — purposefully, not sequentially.

**Bloom's Taxonomy Alignment:**
- Modules 1–2: Remember and Understand (reading tables, identifying features)
- Modules 3–4: Analyze (assessing NPU throughput, mapping pipeline stages)
- Modules 5–6: Evaluate and Create (selecting security features, designing memory layout)
- Final Assessment: Synthesize (producing a complete hardware specification)

### Assessment Strategy

**Formative assessments** (one per subsection, 16 total) use varied formats — multiple choice, true/false, fill-in-the-blank, short answer, scenario decisions, and document review — to maintain engagement and provide immediate feedback. Each formative check is directly aligned to the subsection's learning objective and mirrors the type of task Jin-ho would perform on the job.

**Summative assessment** uses a five-part scenario that requires Jin-ho to produce actual engineering deliverables (written recommendation, pipeline description, assessment paragraph, mapping table, memory allocation table) — the same artifacts he would create in his actual role. This ensures transfer to workplace performance.

### Varied Delivery Formats

Content type drives format selection:
- **Comparative data** (variant tables) → Interactive side-by-side comparison diagrams
- **Architecture** (NPU, pipeline) → Annotated block diagrams with narrated video
- **Decision-making** (variant selection, power modes) → Branching scenario / decision tree
- **Specification analysis** (CSI-2 parameters, package data) → Guided table analysis exercises
- **Planning tasks** (memory layout) → Drag-and-drop allocation exercises
- **Document production** (recommendation, assessment) → Guided writing exercises with structured templates

### Accessibility Considerations

- All video content includes closed captions and full transcripts
- Interactive diagrams include keyboard navigation and screen reader-compatible alt text
- Color is never the sole differentiator in comparison tables (pattern/label redundancy)
- All job aids are provided in both PDF and editable Word/Excel formats
- Formative checks include explanatory feedback for both correct and incorrect responses
